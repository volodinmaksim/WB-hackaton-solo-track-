# Отчёт по решению

## 1. Постановка задачи

Нужно было предсказать `target_1h` — объём отгрузок за последний час.

В `train` были:

- `route_id`
- `timestamp`
- `status_1..6`
- `target_1h`

В `test` были только:

- `id`
- `route_id`
- `timestamp`

Из-за этого финальное решение должно было использовать только те признаки, которые можно честно получить в `test`.

## 2. Ограничения

В финальной модели нельзя было использовать:

- `status_1..6`
- лаги `target_1h`
- любые признаки, которые зависят от будущего

Допустимы были:

- календарные признаки из `timestamp`
- исторические агрегаты, рассчитанные только по `train`

## 3. Финальная рабочая гипотеза

У каждого `route_id` есть собственный устойчивый временной паттерн:

- по времени суток
- по дню недели
- частично по общему календарному контексту

Поэтому итоговая функция прогноза выглядела как:

`f(route_id, timestamp, historical train aggregates) -> target_1h`

## 4. Лучшее решение

Лучшей оказалась версия:

- `LightGBM`
- календарные признаки
- сглаженные исторические агрегаты
- `alpha = 20`

Сильнейшие признаки:

- `target_smooth_route_hour_dow`
- `target_smooth_route_hour`
- `route_id`
- `month`
- `day`

## 5. Метрики

Лучший подтверждённый результат:

- Public leaderboard: `0.3608341250041473`

Лучшая локальная валидация этой версии:

- `0.3835836125379652`

## 6. Что не сработало

### Count-агрегаты

Не дали прироста на public leaderboard.

### Median-агрегаты

Локально заметно ухудшили score.

### Blend `LightGBM + CatBoost`

Локально был очень близок к лучшей версии, но на public leaderboard оказался хуже.

### Слишком точные временные агрегаты

Комбинации вроде:

- `(route_id, hour_slot, day)`
- `(route_id, hour_slot, month)`

оказались слишком шумными и резко ухудшили итог.

## 7. Основной вывод

Для этой задачи решающим оказался не выбор сложной модели, а качество честных признаков.

Лучший результат дал не CatBoost и не ансамбль, а правильно подобранные и сглаженные исторические агрегаты по маршруту и времени.
